# 새 지역/수영장 추가 플레이북

> 성남 유스센터 3곳을 추가하며 확립한 작업 절차. **다른 지역에 반복 적용**할 때 이 문서를 따라간다.
> 마지막 갱신: 2026-07 (성남 = 공사 6곳 + 유스센터 4곳 = 10곳, 수정유스센터는 시범운영 중)

---

## 1. 전체 그림 — 무엇을 건드리나

새 수영장(또는 지역)을 추가하려면 **5개 영역**을 손봐야 한다:

| # | 영역 | 파일 | 하는 일 |
|---|------|------|---------|
| A | **데이터** | `index.html` `POOLS` 배열 | 수영장 항목 추가 (시간표·요금·휴관) |
| B | **표시 등록** | `index.html` `POOL_SHORT`, `BASE_ORDER`, 타임라인 `order` | 리스트/대시보드에 노출 |
| C | **공휴일** | `index.html` `HOLIDAYS`, `LUNAR_HOLIDAYS`, `closedOnHoliday` | 운영자별 공휴일 정책 반영 |
| D | **출처 표기** | `index.html` `.footer-meta` | 곳 수·데이터 출처 갱신 |
| E | **자동 점검** | `scripts/check-schedule.js`, `.github/workflows/*` | 월 1회 크롤링→변경 시 이슈 리포트 |

배포: **GitHub Pages (main 루트 자동 빌드)** — `index.html`을 main에 push하면 1~2분 뒤 반영. (https://aminade.github.io/seongnam-swim/)

## 2. 작업 순서 (체크리스트)

### STEP 0 — 데이터 수집 (크롤링/조사)
- [ ] 지역의 공공 수영장 **운영 주체 파악** (예: OO도시개발공사, OO청소년재단, 구민체육센터…). 주체마다 사이트·공휴일 정책이 다르다.
- [ ] 각 주체 사이트에서 **어느 시설에 수영장(자유수영)이 있는지** 확인. (성남 유스센터는 9곳 중 3곳만 보유)
- [ ] **원본 HTML을 직접 받아서 파싱**한다. `WebFetch`의 요약은 시간표에서 부정확할 때가 많음 → `curl -s -A "Mozilla/5.0 ..." <url>` 후 `수영장 일일이용`/`자유수영` 섹션을 눈으로 확인.
- [ ] 시설별로 수집: **평일/토/일 자유수영 시간대, 요금(성인/청소년/어린이), 정기휴관(매월 N주 일요일), 공휴일 정책, 계절제 여부**.
- [ ] URL 규칙 메모 (예: snyouth.or.kr은 `fmcs/<메인번호>` → 운영시간안내 = `메인번호+13`).

### STEP 1 — `POOLS`에 항목 추가 (§3 스키마 참고)
- [ ] `id`는 지역·주체가 겹치지 않게. (성남 유스센터는 `yc_` 접두사 사용: `yc_yatap` 등)
- [ ] 계절제·요일지정·요금없음 등 특수 케이스는 §3의 확장 필드로 표현.

### STEP 2 — 표시 등록
- [ ] `POOL_SHORT`에 약칭 추가 (대시보드 타임라인 라벨).
- [ ] `renderPools`의 `BASE_ORDER` 배열에 id 추가 (리스트 정렬).
- [ ] `renderCalTimeline`의 `order` 배열에 id 추가 (월별보기 타임라인 행). **두 배열 모두 빠뜨리면 안 됨.**

### STEP 3 — 공휴일 정책 (§4 참고)
- [ ] 운영자의 실제 공휴일 정책을 **원본에서 확인** (그냥 추측 금지).
- [ ] 항목에 `closedOnHoliday:'all'` / `'lunar'` / (없음) 지정.
- [ ] `HOLIDAYS`/`LUNAR_HOLIDAYS`에 해당 연도 공휴일이 정확한지 점검 (대체공휴일 포함).

### STEP 4 — 푸터 출처 갱신
- [ ] 곳 수, 데이터 기준 연월, **출처 도메인**을 최신화.

### STEP 5 — 자동 점검 크롤러 추가 (§5 참고)
- [ ] `check-schedule.js`의 `POOLS_META`(자동반영) 또는 `YOUTH_META`(수동검토)에 메타 추가.
- [ ] `KNOWN`/`KNOWN_YOUTH`에 index.html과 **동일한** 기준값 미러링.
- [ ] 구조가 단순(요일 무관·계절 없음·요금 정형)하면 자동반영, 복잡하면 **탐지·보고 전용**으로.

### STEP 6 — 검증 & 배포
- [ ] **Node 로직 검증**: 슬롯/휴관/공휴일 경계일을 스크립트로 찍어 확인 (§6).
- [ ] **브라우저 프리뷰**: `preview_start` → 리스트·펼침·요금·타임라인·범례·달력 disc 색 확인.
- [ ] `node scripts/check-schedule.js` 로컬 실행 → 새 항목 "이상 없음" 확인 (기준값과 라이브가 일치해야 함).
- [ ] `index.html`(+크롤러) 커밋 → main push → Pages 반영 확인 (`curl`로 라이브 grep).
- [ ] `gh workflow run update-schedule.yml`으로 CI 1회 검증 → 오탐 이슈 없이 success인지 확인.

## 3. 데이터 모델 스펙 — `POOLS` 항목

```js
{
  id:'yc_yatap',                 // 고유 id (지역·주체 접두사 권장)
  name:'야탑유스센터',            // 표시 이름
  area:'분당구 야탑동',           // 위치 (구+동, 불명확하면 구까지만)
  url:'https://www.snyouth.or.kr/fmcs/158',  // 상세 페이지
  closedWeeks:[1,3],             // 매월 N번째 일요일 정기휴관
  closedOnHoliday:'all',         // 공휴일 정책 (§4). 없으면 공휴일에도 운영
  feature:'유스센터',            // 메타 뱃지(선택). 없으면 null
  weekdaySlots:[...], saturdaySlots:..., sundaySlots:[...],
  weekdayPrice:{adult,teen,child}|null,
  weekendPrice:{adult,teen,child}|null,
  priceNote:'요금 정보 없음'      // price가 null일 때 표시할 문구(선택)
}
```

### 슬롯 형식
```js
{ time:'HH:MM~HH:MM', dur:<분>, days?:[1,3,5] }
```
- `dur`는 분 단위 정수. 임의값 허용 (평일 50분, 주말 90분, 기존 60/120 등).
  - 타임라인 색: `dur<=60` 연한, `<=90` 중간, 그 외 진한 (2시간).
- `days`: 특정 요일만 운영하는 슬롯 (getDay 값, 1=월…5=금). 예: 중원 "월·수·금만 15:00" → `days:[1,3,5]`. **평일 슬롯에만 적용됨** (`slotsFor`가 필터).

### 계절제 주말 (하절기/동절기)
`saturdaySlots`(또는 sunday)를 배열 대신 객체로:
```js
saturdaySlots:{ summer:[...], winter:[...] }
```
- `isSummer(date)` = 3~10월(하절기), 11~2월(동절기). `resolveSlots`가 자동 선택.
- 배열로 주면 계절 무관 동일.

### 요금 없음 / 무료
- `weekdayPrice:null, weekendPrice:null` → 펼침 시 `priceNote`(기본 '요금 정보 없음') 한 줄 표시.
- 부분 정보(예: 청소년 무료)는 `priceNote:'청소년 무료 · 그 외 요금 정보 없음'`처럼.

## 4. 공휴일 로직 — 운영자별로 다르다 (중요)

핵심 교훈: **운영 주체마다 공휴일 정책이 다르므로 원본에서 확인해야 한다.**

| 정책 | `closedOnHoliday` | 성남 사례 | 근거 |
|------|-------------------|-----------|------|
| 모든 법정공휴일 휴관 | `'all'` | 유스센터 3곳 | "매월 N주 일요일 **및 법정 공휴일**" |
| 설·추석 연휴만 휴장 | `'lunar'` | 도시개발공사 6곳 | "매월 둘째·넷째 일요일 **및 설·추석 연휴 휴장**" (그 외 공휴일은 '일·공휴일' 시간표로 운영) |
| 공휴일에도 운영 | (필드 없음) | — | — |

- `HOLIDAYS`: 그 해 **모든 법정공휴일** 날짜 (`YYYY-MM-DD`). **대체공휴일 포함**해야 함.
- `LUNAR_HOLIDAYS`: 설·추석 연휴 날짜만 (공사류 휴장 판단용).
- `isClosedDay(pool,date)`가 두 정책 + 정기휴관(일요일)을 종합.
- 공휴일엔 `slotsFor`가 **일요일 시간표**를 반환한다 (운영하는 경우).

### 연도 넘어갈 때 / 새 지역
- 매년 `HOLIDAYS`·`LUNAR_HOLIDAYS`를 그 해 기준으로 갱신. 앱은 **이번달+다음달**만 보여주므로 최소한 그 범위는 정확해야 한다.
- **대체공휴일 규칙**: 설·추석·삼일절·어린이날·부처님오신날·광복절·개천절·한글날·성탄절이 주말과 겹치면 다음 평일로 대체. (신정·현충일은 대체 없음)
- 제헌절(7/17)은 2026년부터 공휴일로 재지정됨 — 연도별 관보 확인.

## 5. 자동 점검 크롤러 (`scripts/check-schedule.js`)

한 달에 한 번(매월 28일·1일 cron) 라이브 사이트를 크롤링해 **하드코딩된 기준값과 비교** → 변경 시 GitHub 이슈로 리포트.

### 구조
- `POOLS_META` + `KNOWN` : 도시개발공사류 (자동반영 대상, `changed`)
- `YOUTH_META` + `KNOWN_YOUTH` : 유스센터류 (수동검토, `youthChanged`)
- `fetchText(url, tries=3)` : **재시도 포함 fetch** — 일시적 실패로 오탐 이슈가 생기지 않게 함. (이것 없으면 네트워크 블립마다 빈 이슈 생성됨)
- 추출→파싱→`diff`/`diffYouth`(기준값과 슬롯 집합·휴관주차 비교)→ 결과 JSON.
- 종료코드: `2`=변경, `1`=크롤 실패, `0`=이상없음. 워크플로는 `!=0`일 때 이슈 생성.

### 자동반영 vs 탐지·보고 — 선택 기준
- **자동반영**(`/confirm` 시 `apply-changes.js`가 index.html 수정): closedWeeks/요금/평일슬롯이 **단순 정형**일 때만. → `changed`, `CHANGES_JSON`에 포함.
- **탐지·보고 전용**: 계절제·요일지정·null요금 등 구조가 복잡하면 자동반영이 위험 → `youthChanged`로 분리, `CHANGES_JSON`에 **넣지 않음**. 이슈엔 "수동 반영" 섹션으로 안내, 사람이 index.html 직접 수정.

### 파싱 함정 (겪은 것들)
- **섹션 경계어 오절단**: 표 캡션("시설 및 안내", "대표전화")이 본문보다 앞에 나와 섹션이 잘림 → 뒤에 안정적으로 오는 경계어("이용수칙","공지사항","라이선스","공공누리")만 쓰고 **길이 상한(4000자)**을 둠.
- **시간대 헤더 오탐**: "토요일 06:30~19:00" 같은 운영범위(수백 분)는 슬롯이 아님 → **duration 40~120분만** 슬롯으로 인정.
- **휴관주차 표기 두 형태**: 숫자("매월 1, 3주") + 한글 서수("첫째주, 셋째주") 둘 다 파싱.
- **기준값 미러링**: `KNOWN_YOUTH`의 slots는 index.html의 **모든 요일 슬롯 시간의 합집합**. 페이지에서 뽑은 전체 시간집합과 비교(누락/초과)한다.

### 워크플로 2종
- `update-schedule.yml`: cron/수동 → 크롤러 실행 → 변경 시 이슈(제목 `[자동] …변경 감지`, 라벨 `schedule-update`). 본문에 공사(자동반영)/유스센터(수동) 섹션 분리, `CHANGES_JSON` 주석에 공사만.
- `apply-schedule.yml`: 이슈 댓글 `/confirm`(소유자만) → `apply-changes.js` → 커밋 → Pages 배포. `/reject` → 이슈 닫기.

## 6. 검증 방법 (반복 활용)

### Node 로직 스냅샷 (파일에서 함수만 뽑아 테스트)
`index.html`은 브라우저 코드라 통째로 실행 불가 → `POOLS`/`HOLIDAYS`/함수 블록을 정규식으로 잘라 임시 `.mjs`로 실행하면 슬롯/휴관/공휴일 경계일을 확정적으로 검증할 수 있다. (이번에 공휴일 경계일 7/17·8/15·설날·추석·대체공휴일을 이렇게 표로 찍어 확인했다.)

### 브라우저 프리뷰
- `.claude/launch.json` (python http.server). **주의: 이 머신엔 homebrew python이 없어 `/usr/bin/python3` 사용, 포트 충돌 시 `autoPort:true`.**
- 확인 항목: 리스트 노출·정렬, 항목 펼침(슬롯 pill·요금/요금없음 문구), 월별보기 타임라인(행 추가·바 색 3단계·범례), 달력 disc 색(공휴일 경계일), 탭/월 이동 리셋.

### 크롤러
- `node scripts/check-schedule.js` → 새 항목 "이상 없음" (기준값=라이브면 정상).
- `gh workflow run update-schedule.yml` → CI에서 success + 오탐 이슈 0.

## 7. 다지역 확장 시 고려사항 (앞으로)

현재 구조는 **단일 지역(성남) 전제**다. 여러 지역으로 늘리면:
- `POOLS` 항목에 `region` 필드를 추가하고, UI에 지역 선택/그룹핑을 도입할지 결정 필요.
- `POOL_SHORT`/`BASE_ORDER`/타임라인 `order`가 지역별로 커지므로 **자동 파생**(POOLS에서 생성)으로 리팩터 고려.
- 공휴일은 전국 공통(법정공휴일)이라 `HOLIDAYS`는 재사용, 단 **운영자별 정책**(`closedOnHoliday`)은 지역·주체마다 다시 확인.
- 크롤러 `META`/`KNOWN`이 커지므로 지역별 모듈 분리 고려.
- 푸터 출처가 여러 개 → 목록화.

지금 당장은 위 §2 체크리스트대로 **항목만 추가**하면 되고, 항목이 수십 개를 넘어가면 위 리팩터를 검토한다.

## 부록 — 성남에서 확인된 사실 (참고 데이터)
- 유스센터 URL: 야탑 `fmcs/158`(메인 145), 중원 `fmcs/57`(44), 판교 `fmcs/133`(120), 수정 `fmcs/32`(메인 19, 아직 시간표 미반영). 운영시간안내 = 메인+13.
- 수영장 보유: 야탑·중원·판교·수정 (10곳 중 4곳). 서현·정자·양지·은행은 없음.
- 유스센터 주말: 하절기 6부제 / 동절기 5부제 (각 90분). 평일 50분.
- 요금: 유스센터 초2이하 1,900 / 청소년 2,500 / 성인 3,000 (야탑은 페이지 미표기, 중원 청소년 무료).
- 공사 6곳: 일반 공휴일 운영(일·공휴일 시간표), 설·추석만 휴장.

### 시범운영 중인 신규 시설 — 수정유스센터 (2026-07)
정식 운영시간안내 페이지(`fmcs/32`)에 아직 시간표가 반영되지 않은, **표준 패턴에서 벗어난 케이스**. 다음 지역에서 비슷한 "공사 중·시범운영" 시설을 만나면 참고:
- 근거는 정식 페이지가 아니라 **공지사항 게시판의 개별 글**(`fmcs/22?action=read&action-value=<hash>`). 게시판 목록 URL(`fmcs/22`)에서 `action-value=<32자리 hex>` 패턴으로 고정 공지 글의 permalink를 찾을 수 있음(관리사이트 `mngcms/` URL은 공개 URL이 아니므로 쓰지 말 것 — `fmcs/<번호>` 공개 게시판에서 동일 글의 `action-value`를 찾아 사용).
- 데이터 모델: `POOLS` 항목에 `startsOn:'YYYY-MM-DD'` 필드 추가 → `isClosedDay`가 그 날짜 이전을 전부 휴관으로 처리. **트라이얼 시작일 이전에 "운영중"으로 잘못 표시되는 버그를 막기 위해 필수** (이 필드가 없으면 앱이 시작일과 무관하게 요일별 슬롯만 보고 운영중 판정을 내림).
- 크롤러(`scripts/check-schedule.js`): 표준 `extractYouthSwimSection` 파싱이 안 통하므로 별도 `SUJEONG_NOTICE`/`crawlSujeongNotice`/`diffSujeongNotice` 추가. 게시판의 고정 공지 `action-value`가 바뀌면(=공지 교체) 수동 검토 플래그, 정식 페이지에 실제 시간표(슬롯)가 등장하면 "정식 운영 전환" 플래그. 둘 다 `youthResults`에 합류해 기존 리포트/이슈 파이프라인을 그대로 탄다(워크플로 YAML 수정 불필요).
- 정식 운영 전환 시 할 일: `SUJEONG_NOTICE`를 제거하고 `YOUTH_META`+`KNOWN_YOUTH`로 승격, `index.html`에서 `startsOn` 제거하고 실제 시간표로 교체, `priceNote`/`feature`의 "시범운영" 문구 정리.
